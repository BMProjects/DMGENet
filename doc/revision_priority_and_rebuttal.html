DMGENet 修稿优先级清单 + Rebuttal 回复模板

这份网页文档把当前项目核查出的论文与代码问题，整理成可执行的修稿优先级、 reviewer 对应动作、以及可直接套用的 rebuttal 模板。目标不是“全面重做”，而是在尽量短时间内先修掉高风险问题，再补最能打动审稿人的证据链。

期刊状态: Major Revision 当前项目: DMGENet 核心目标: 快速提高可信度与可接受性
总览 修稿优先级 审稿意见对照 Rebuttal 模板 写作注意事项

一页总览

P0 必须先改 先处理会影响可信度的问题

测试集泄漏、论文与代码实验设置不一致、 ASTAM / RLMC 描述与实现偏差、标题摘要关键词格式违规。这些问题不先修，后面的补实验很难真正起作用。

P1 高收益补强 补足审稿人最在意的证据

补 12h / 24h 结果、补更强 baseline、补 mean ± std 与显著性、补硬件和训练协议、补效率分析与 limitations。

P2 有时间再做 进一步提升说服力

新数据集泛化、噪声鲁棒性、漂移统计检验、更多阈值和超参数敏感性、更多可解释性图表。这些加分明显，但不一定是短期硬门槛。

建议修稿冲刺顺序

1
先统一事实版本

把论文里所有“窗口长度、预测步长、标准化方式、图记号、模块公式”改成与最终实验脚本一致。

2
修掉评估漏洞

明确验证集训练 Actor、测试集只报告最终结果，避免任何“用 test 选模型”的流程。

3
补最关键实验

优先整理 1h / 6h / 12h / 24h，以及更强 baseline、ablation、ensemble 对比。

4
补可复现信息

把硬件、软件版本、超参数、随机种子、训练轮数、指标统计都写完整。

5
最后润色 rebuttal

把每条意见分成“承认问题 / 已做修改 / 修改位置 / 残余限制”四部分，语气平稳、具体、可核查。

修稿优先级清单

下面这张表按“风险大小”和“短时间内的收益”排序。建议修稿时严格按这张表推进，不要一上来就先补文献综述或美化图表。

优先级 问题 论文修改动作 代码 / 实验修改动作 预期收益
P0 测试集泄漏 明确说明 validation 用于 RL 训练与模型选择， test 仅作最终评估；重写实验协议段落。 取消基于 test loss 选最佳 Actor 和最佳 run 的流程，改为基于 validation 选择。 直接修复最严重的方法学漏洞，是 rebuttal 的第一优先项。
P0 论文与代码设置不一致 统一窗口长度、预测步长、标准化方式、图记号、功能图命名、ASTAM 公式描述。 确定最终采用的真实配置后，只保留对应版本的实验结果和脚本。 防止 reviewer 继续抓“技术不一致”和“复现不可信”。
P0 标题 / 摘要 / 关键词格式违规 去除标题和关键词中的缩写，摘要中所有缩写首次完整定义。 无核心代码动作，但需要统一稿件与投稿系统内容。 这是编辑部硬规范，不改可能直接卡流程。
P1 12h / 24h 结果与更长 horizon 论证不足 在主结果表与讨论中突出 12h / 24h，解释多步误差累积规律。 整理现有 12 / 24 小时实验；若可能，补 1 / 6 / 12 / 24 四档统一对照。 直接回应 Reviewer #1 对行业标准 horizon 的质疑。
P1 baseline 不够强 补最新且相关的 air quality forecasting 对比，给出 baseline 选择理由。 优先补可快速实现或可复用公开实现的强基线，至少补静态平均集成与固定权重集成。 这是三位 reviewer 的共识问题，补上后整体说服力会上一个台阶。
P1 统计显著性与方差缺失 所有核心表格改为 mean ± std，必要时补 t-test 或 Wilcoxon 说明。 基础模型和 RLMC 都按多随机种子重复运行并保存统计结果。 能显著缓解“结果偶然性”质疑。
P1 复现信息不足 新增 Implementation Details / Reproducibility 小节。 补硬件、 CUDA / PyTorch 版本、 batch size、随机种子、训练轮数、学习率策略、参数量与耗时。 这类信息补起来快，但 reviewer 非常看重。
P1 Discussion / Limitations 太弱 新增 limitations 小节，结论中给出具体数字， discussion 解释各图的贡献差异。 从现有消融与单图结果里提炼“为何 N / F 图更关键”等证据。 对 Reviewer #3 的满足度提升很快。
P2 数据过旧与泛化问题 若无法补新数据，必须诚实写入 limitations，并弱化“现实部署”措辞。 有条件时新增一个较新的城市数据集或时间段做外部验证。 高价值，但时间成本最高。
P2 鲁棒性 / 漂移分析缺失 补 non-stationarity 讨论、噪声与缺失值鲁棒性实验。 增加输入扰动、滚动均值方差、简单漂移统计脚本。 能进一步回应 Reviewer #1 / #2，但不是首要门槛。
建议实际执行时把全部工作分成两轮。第一轮只做 P0 + 最有把握的 P1，先确保“方法学没硬伤”；第二轮再补高成本实验和更多图表。

审稿意见对照行动板

这一部分适合在修稿时做任务分派。每条 reviewer 意见都尽量拆成“论文动作”和“代码动作”，避免出现“论文想改但实验跟不上”或者“代码改了但文稿没跟”的情况。

Reviewer #1

问题 最短路径回应 建议落点
数据集较旧 最佳方案是补新数据；若时间不够，明确承认限制，并把“practical utility”降级成“validated on this benchmark”。 Introduction, Discussion, Limitations, Conclusion
预测 horizon 太短 突出 12h / 24h 结果，主表按 1 / 6 / 12 / 24 或完整 1-24h 展示。 Main results table, Discussion
缺少强 baseline 补近两年强相关模型，至少说明为什么选这些 baseline。 Related Work, Baselines, Results
图记号不统一 统一 D / N / S / F，避免 POI / F 混用。 Figures, equations, text, table captions
未讨论 distribution drift 补非平稳性讨论和简单统计分析，至少说明模型如何缓解而非彻底解决漂移。 Method motivation, Discussion
MDP 状态长度和 Markov 假设 先把论文里的状态长度改成真实实验设置，再明确说这是近似 MDP，而不是严格成立。 RLMC section, Limitations
超参数选择显得启发式 补 sensitivity study，比直接做复杂元启发式优化更省时。 Appendix or extra experiment subsection
未说明为何 RL ensemble 优于多任务学习 在引言和讨论中明确“动态样本级权重分配”是 RL ensemble 的主要优势。 Introduction, Discussion

Reviewer #2

问题 最短路径回应 建议落点
novelty 不够清楚 不要把每个子模块都写成全新，而要强调“多图建模 + 时空异质性建模 + 动态强化集成”的组合创新。 Abstract, Introduction, Contributions
理论支撑不足 补每个设计选择的动机和假设边界，尤其是 ASTAM、RLMC、图构建。 Method section
对比不充分 补 SOTA，并说明实现来源、参数公平性和训练协议。 Baselines and Results
claim 过强 删掉 unsupported claims，把“robust / generalizable / practical”改成更谨慎的 benchmark 范围表述。 Abstract, Conclusion
可复现性不足 单列一节写 implementation details，并公开脚本结构。 Implementation details
统计显著性缺失 统一给 mean ± std，补 significance note。 All result tables

问题 最短路径回应 建议落点

写作注意事项

回复时尽量避免
  • 不要说 “the reviewer misunderstood”. 用 “we have clarified” 更稳妥。
  • 不要硬说 “this does not affect the conclusion” 除非你们真的重跑并验证过。
  • 不要把所有问题都写成“future work”，尤其是测试协议和格式规范类问题。
  • 不要把“当前代码支持”直接等同于“论文已证明”，必须给表格或文字证据。
回复时建议坚持
  • 每条意见都按 “感谢 + 承认 / 澄清 + 已改内容 + 修改位置” 的结构来写。
  • 如果没有完全满足，也要诚实写残余限制和当前 revision 的边界。
  • 所有新增实验都要写清随机种子、数据划分、评价指标和硬件环境。
  • 结论和摘要都要收敛到“在本 benchmark 上”的证据范围内。
